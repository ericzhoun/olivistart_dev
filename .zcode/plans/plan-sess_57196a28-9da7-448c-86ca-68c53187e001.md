# Plan: Account Management + Student Artwork Photos

## Overview
Add two capabilities to `account.html`:
1. **Account management**: change password (via email code), edit contact info, add/manage student profiles.
2. **Children's artwork photos**: parents and admin upload photos, tagged to a student; parents view a gallery of their children's work.

## Key constraints discovered
- **No "change password with current password" endpoint** on Butterbase. Auth only exposes `forgot-password` (emails a 6-digit code) → `reset-password` (code + new password). So "change password" on the account page drives this email-code flow in-page.
- **Storage access control**: end-users can only download files *they* uploaded. Since admin (Olivia) also uploads artwork for a child, the client can't fetch those directly. A serverless function mints presigned download URLs using `SERVICE_KEY` and gates access via the `artwork_photos` metadata table + RLS.
- Existing conventions: functions are `handler(req, ctx)` with `auth: "required"`, use `ctx.db.query(sql, params)`, `ctx.user.id`, return `new Response(JSON.stringify(...))`. Deployed via `backend/deploy.sh` CONFIGS array.

## Phase 1 — Schema (new tables)

Apply via `manage_schema` action "apply" (additive — no `_drop`).

### `students` table
| column | type | notes |
|---|---|---|
| id | uuid PK | `gen_random_uuid()` |
| user_id | uuid | owner = parent. Nullable so admin-created rows (for a parent's child) can be claimed, matching the enrollments guest pattern. |
| name | text | child's name |
| age | text | (text to match existing `enrollments.child_age`) |
| dob | text | date of birth |
| notes | text | optional (allergies, interests) |
| created_at | timestamptz | `now()` |

### `artwork_photos` table
| column | type | notes |
|---|---|---|
| id | uuid PK | |
| student_id | uuid FK→students | `onDelete: CASCADE` |
| storage_object_id | text | the Butterbase `objectId` (durable ref; never a URL) |
| caption | text | optional |
| uploaded_by | text | "parent" or "admin" |
| created_at | timestamptz | `now()` |

## Phase 2 — RLS

- **`students`**: user isolation on `user_id` (SELECT/INSERT/UPDATE/DELETE) via `create_user_isolation`, plus a service-bypass (automatic). This lets a parent manage only their own children.
- **`artwork_photos`**: parent can SELECT rows where the photo's `student_id` belongs to one of *their* students. Implement with a custom SELECT policy using an `EXISTS(SELECT 1 FROM students WHERE id = artwork_photos.student_id AND students.user_id = current_user_id()::uuid)` subquery. INSERT/DELETE gated through a function (server-side, service role) so the parent never needs direct write — the function validates ownership and inserts with `SERVICE_KEY`. Service bypass is automatic.

## Phase 3 — Backend functions (3 new)

All `auth: "required"`, `allow_service_key_impersonation: false`, deployed via `deploy.sh`.

### `manage-account` (POST `/manage-account`)
Single function for account operations to keep the surface small:
- `action: "update-contact"` — updates editable fields on the user's enrollments (parent_name, student_phone, emergency_contact, allergies). Writes via `ctx.db` as the user (RLS-enforced: only own enrollments).
- `action: "change-password-init"` — calls `POST /auth/{app}/forgot-password` with the user's email server-side, returns `{ sent: true }`. (Backend makes the call so the email is always the authenticated user's email — the client never chooses the target email.)
- `action: "change-password-confirm"` — calls `POST /auth/{app}/reset-password` with `{ email, code, new_password }`, returns success/failure.

### `manage-students` (POST `/manage-students`)
- `action: "list"` — `SELECT * FROM students WHERE user_id = $1` (RLS would also enforce; belt-and-suspenders).
- `action: "add"` — insert a student row owned by `ctx.user.id`.
- `action: "update"` — update name/age/dob/notes on a student the caller owns.
- `action: "delete"` — delete a student the caller owns (cascades artwork_photos).

### `manage-artwork` (POST `/manage-artwork`)
Handles the storage-presigned-URL lifecycle, validated server-side:
- `action: "upload-url"` — input `{ student_id, filename, content_type, size_bytes }`. Verify the student belongs to `ctx.user.id` (or admin). Call `POST /storage/{app}/upload` with `SERVICE_KEY` to get `{ uploadUrl, objectId }`. Insert an `artwork_photos` row (uploaded_by "parent"/"admin"). Return `{ upload_url, object_id }`. Client then PUTs the file bytes.
- `action: "download-urls"` — input `{ photo_ids: [...] }`. Query `artwork_photos` joined to `students` filtered by `ctx.user.id` (RLS-enforced) — rows the parent can't see simply aren't returned. For each visible photo, call `GET /storage/{app}/download/{object_id}` with `SERVICE_KEY` to mint a short-lived download URL. Return `{ photos: [{ id, caption, uploaded_by, created_at, download_url }] }`.
- `action: "delete"` — verify ownership, delete the storage object (SERVICE_KEY) and the metadata row.

Admin (Olivia) detection: same `isAdmin()` email check used elsewhere; the function checks `ctx.user` email against the admin allowlist and allows admin to upload/view for any student.

## Phase 4 — Frontend: `js/account.js` + `account.html`

The existing `account.js` renders the whole page into `#account-root`. I'll restructure the render into **tabs/sections** without breaking the existing enrollment flow:

1. **"My Enrollments"** (existing — unchanged, stays at top).
2. **"Profile & Security"** (new section):
   - Contact info form (parent_name, phone, emergency contact, allergies) — edits apply to the user's enrollments.
   - Change password: a 2-step in-page form. Step 1 "Send code to my email" → calls `manage-account` change-password-init. Step 2 enters the 6-digit code + new password → change-password-confirm. On success, force re-login.
3. **"Students"** (new section):
   - List of student profile cards (name, age, DOB, notes) with edit/delete.
   - "Add Student" button → inline form.
4. **"Artwork"** (new section):
   - For each student, a thumbnail gallery. "Upload photo" button per student → file picker → upload-url → PUT → refresh.
   - Thumbnails open a lightbox (reuse the `portfolio-lightbox` dialog pattern from `portfolio.html`/`portfolio.js`).
   - Download URLs resolved in parallel via `manage-artwork` download-urls (1-hour TTL).

New helpers added to `js/api.js`: none needed — `callFunction` already covers function calls. A small `uploadFileToStorage(uploadUrl, file)` and `escapeHtml` helper go in `account.js`.

`account.html`: add the `<dialog>` lightbox markup (matching portfolio.html) since the page is JS-rendered.

## Phase 5 — CSS (`css/style.css`)

Add styles for: account tabs/section headers, student cards, artwork gallery grid + thumbnails, change-password step UI. Reuse existing tokens (`--color-surface`, `--color-border`, `--radius`, `.btn`, `.btn-sm`, `.auth-error`).
